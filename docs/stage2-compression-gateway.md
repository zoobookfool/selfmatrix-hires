# Stage 2 設計文書 — 圧縮ゲートウェイ

2026-07-06 起草。requirements.md §4.1 の「要検証の設計仮説」を検証した結果としての設計文書であり、requirements.md §9 の未解決課題「圧縮ゲートウェイの実現可能性」に対する回答を兼ねる。requirements.md §7 Stage 2 exit criteria の「JackTrip の UDP ペイロード構造と圧縮データの互換性を検証済み(挟み込みが技術的に成立するか、実質的に別トランスポートの自作が必要かを判定し、設計を確定させている)」を満たすことを目的とする。§7 PoC 計画を除いて実装は行っておらず、数値は実測または算出式に基づく見積りであり、実装未了の箇所は「見込み」「想定」と明記する。

## 1. 目的と位置づけ

requirements.md §4.1 は、可逆圧縮伝送の実現方式について次の設計仮説を「要検証」としていた。

> JackTrip の UDP 音声プロトコルはヘッダ(`DefaultHeader`、16byte)に `SamplingRate`/`BitResolution`/`BufferSize` 等のフィールドを持ち、ペイロードは実サンプル数を前提にした固定フォーマット(生PCMサンプル値)であるため、WavPack で圧縮したバイト列をこのペイロード部にそのまま「挟む」ことがプロトコル上成立するのか、それとも JackTrip の外側に TCP/UDP ソケットを新設した実質的な別トランスポートを自作する必要があるのかは技術的に未検証である。

本文書はこの仮説を、JackTrip のソースコード調査(UdpHubListener.cpp、JackTripWorker.cpp、PacketHeader.{h,cpp}、UdpDataProtocol.cpp、JitterBuffer.cpp、Settings.cpp、jacktrip_globals.h)および WavPack 5.9.0 の実測ベンチマークに基づいて検証した結果である。結論を先取りすると、「ペイロードに直接圧縮データを挟む」案は不成立であり、「JackTrip 無改造 + 外側 UDP プロキシによる透過トンネル圧縮」案が技術的に成立する(詳細は §2)。

## 2. 設計判断: 挟み込み案 vs 透過トンネル案

### 2.1 案A: JackTrip のペイロードに直接圧縮データを挟む

requirements.md §4.1 が当初「第一候補」としていた実現方式。JackTrip プロセス自体、または JackTrip に密結合したラッパーが、UDP パケットのペイロード部だけを圧縮データに置き換える案である。

**不成立と判断する根拠**:

- PacketHeader.cpp の `validatePeerHeader()` は、ヘッダの `BufferSize`・チャンネル数・`BitResolution` から `hdr_size + BufferSize × channels × (bitres/8)` という期待パケット長を逆算し、これと実際の受信バイト数を照合する(自己記述的検証)。圧縮によってペイロード長が可変になると、この検証式が成立しなくなり、受信側がサイズ不足・不正パケットとして扱う。
- ペイロード自体は「チャンネルごとに連続したサンプル配列(非インターリーブ)」という固定レイアウトの生PCMを前提としており、圧縮コーデックの可変長出力(WavPack のブロックはヘッダ・状態を含む可変長バイト列)をそのまま代入できる構造になっていない。
- この検証・デコード処理は JackTrip 本体のコード(`UdpDataProtocol.cpp`)に組み込まれているため、圧縮を持ち込むには JackTrip 本体の改修が必要になる。これは「JackTrip 無改造」という運用上望ましい制約(バージョン追従の容易さ、AGPL-3.0系ライセンスの疑義が残る本体への手を加えない設計、requirements.md §6 のライセンス留保)と相容れない。

以上より、案Aは「ヘッダの自己記述的検証と衝突する」「JackTrip 本体の改修が要る」という2点で不採用と判断する。

### 2.2 案B: 透過トンネル型圧縮プロキシ(JackTrip 無改造)

JackTrip 本体は一切変更せず、クライアント側・VPS側それぞれにユーザ空間の UDP プロキシデーモンを常駐させ、JackTrip がプロキシへ平文で送受信したパケットを、プロキシがヘッダ部を素通し・ペイロード部のみ圧縮した上で外側の圧縮トンネルへ中継する案。

**成立すると判断する根拠**:

1. hub(UdpHubListener.cpp の `getPoolID()`)は「サーバー側UDPポート番号」と「送信元(addr, port)」の一致だけでクライアントを識別し、ペイロード内容は一切見ない。プロキシが送信元をlocalhostに書き換えても、クライアントごとに専用ポートへの1:1中継を維持すれば識別ロジックは壊れない。
2. サーバー割当UDPポート(`mBasePort + id`、後述§3)はTCP 4464のハンドシェイクで平文4byte little-endian整数として明示的に送られてくる。プロキシがこのTCPストリームをバイト単位で完全に透過中継しつつパッシブに覗き見ることで、動的ポートを確実に学習できる。静的なポート範囲(61002-62000)を丸ごと転送するだけでは、複数クライアント混在時に「どのUDPポートがどのJackTripクライアント宛か」を区別する情報が欠落するため、この動的学習が必須である。
3. クライアント→hubの送信元UDPポートは、hub が最初のパケットの実送信元から自動学習する仕様(NAT越え対応、JackTripWorker.cpp の `receivedFirstPacketUDP`)であるため、クライアント側プロキシがローカルで別ポートを使っても支障ない。
4. パケットヘッダ(16byte固定・自己記述フィールド)からペイロード長を一意に算出できるため、プロキシは「ヘッダを読んでペイロード長を検算し、ペイロード部分だけ可逆圧縮する」実装が可能。ヘッダ自体(TimeStamp/SeqNumberを含む)は素通しが必須。

以上より、**案B(透過トンネル型圧縮プロキシ、JackTrip 無改造)を Stage 2 の採用案とする**。

### 2.3 比較まとめ

| 観点 | 案A: ペイロード直接挟み込み | 案B: 透過トンネルプロキシ(採用) |
|---|---|---|
| JackTrip 本体改修 | 必要(ヘッダ検証式の変更が要る) | 不要 |
| ヘッダの自己記述検証との整合 | 不整合(可変長ペイロードと衝突) | 整合(ヘッダは素通し、ペイロードのみ圧縮) |
| ライセンス(AGPL-3.0系疑義)への影響 | 本体組み込みのため影響を受けうる | 別プロセスのため影響を受けない見込み(requirements.md §6 と同じ留保付き) |
| バージョン追従 | JackTrip のマイナー更新ごとに追随改修が要る | ハンドシェイク仕様(TCP 4464のポート通知形式)が変わらない限り追随不要 |
| 実装対象 | JackTrip フォーク | 新規プロキシデーモン(クライアント側・VPS側) |

## 3. アーキテクチャ

### 3.1 構成図

```
[クライアントPC]                              [VPS]
┌───────────────────┐                 ┌───────────────────┐
│ JackTrip client    │                 │ JackTrip hub (-S)  │
│ (無改造)            │                 │ (無改造)            │
│  TCP 4464 ─┐        │                 │        ┌─ TCP 4464 │
│  UDP local ─┤        │                 │        ├─ UDP     │
└─────┼──────┘        │                 │        └───┼──────┘
      │(localhost)     │                 │  (localhost)│
┌─────┴──────────────┐│                 │┌────────────┴─────┐
│ クライアント側        ││                 ││ VPS側              │
│ 圧縮プロキシ           ││   圧縮トンネル    ││ 圧縮プロキシ         │
│ - TCP 4464 透過中継   │◀════════════════▶│ - TCP 4464 透過中継 │
```

### 3.2 コンポーネント配置

- **クライアント側**: JackTrip クライアント本体(無改造)は、起動パラメータのピアアドレスのみを変更し、localhost 上のクライアント側プロキシへ平文で接続する。プロキシがWAN側の実際の通信(VPSの圧縮プロキシ)を担う。
- **VPS側**: JackTrip hub(無改造)も同様に、localhost 上のVPS側プロキシとの間で平文送受信を行う。hub 自体は自分がプロキシ越しに話していることを認識しない。
- ポート/プロトコル自体は変更しない。変わるのは「JackTrip が直接話す相手」がプロキシのlocalhostアドレスになる点のみ。

### 3.3 ポート/セッションの扱い

- VPS側のサーバー割当UDPポートは `mBasePort + id` で決まり、`mBasePort = 61002 + (server_port - 4464)`(既定のTCPポート4464の場合は61002起点)、`id` は接続クライアントに割り当てられる空きスロット番号(最大 `gMaxThreads = 1024`)である。このポート番号はTCP 4464ハンドシェイク内で4byte little-endian整数として平文送出される(`sendUdpPort`/`readClientUdpPort` に相当する処理)。
- VPS側プロキシは、TCP 4464のストリームをバイト単位で完全に透過中継しながら、この4byte整数をパッシブに読み取り、「TCPセッション(＝どのJackTripクライアントか) ⇔ 割当UDPポート」の対応表を構築する。これにより、静的なポート範囲(61002-62000)を丸ごと転送するのではなく、クライアントごとに1:1のUDPポートペアリングを動的に張ることができる。
- クライアント側の送信元UDPポートはTCPで事前通知されない仕様(NAT越えのための意図的設計)のため、hub 側は最初に受信したUDPパケットの実送信元ポートから自動学習する。この学習プロセスはプロキシを挟んでも壊れない(プロキシのlocalhostソケットが新しい「送信元」になるだけで、1:1中継である限り識別ロジックは影響を受けない)。
- セッションのタイムアウトは `gClientGoneTimeoutMs = 10000`(10秒。この間隔パケット未着でセッション終了)と `gTimeOutMultiThreadedServer = 10000`(新規接続の最初のUDPパケットが10秒以内に届かないとhubがworkerを解放)の2つが関係する。プロキシの再起動やハングは、この10秒以内に復旧しなければJackTrip側のセッションが切断される制約として扱う。

**バイトレベル確定(2026-07-06、JackTrip mainブランチのソース(UdpHubListener/JackTripWorker/JackTrip/Auth/PacketHeader各.cpp/.h)を直接読了して確認)**:

以下の箇条書きのうち、バイト列オフセット・フィールド長など直接読了で確定した事実(次項および§3.3訂正部分)は推測を含まない。ただし二段階UDP受信の意図・タイミング(下記3項目目)は読了未了のソースに依存し未確認である旨をその項目内に明記しているので、確度はその記述に従うこと(一括して「推測なし」ではない)。

- 平文モードのTCPバイト列は、C→S: 68byte = `[0-3: LE int32 mReceiverBindPort][4-67: リモートクライアント名 UTF-8+NUL、64byte固定]`、S→C: 4byte = `LE int32 サーバー割当UDPポート(mBasePort+id)`、で確定した。上記の「4byte little-endian整数」という記述はこの調査により裏付けられた。
- **重要な訂正**: C→Sで送られる4byteのポート値(`mReceiverBindPort`)は、hub側では実UDP宛先ポートとして一切採用されない。`UdpHubListener::readyRead()` は `setJackTrip(id, PeerAddress, mBasePort+id, 0, ...)` と client_port に明示的に `0` を渡しており、実際のクライアントUDP送信元は `JackTripWorker::receivedFirstPacketUDP()` が受信した最初のUDPパケットの実送信元ポート(OSカーネルが解決した値)から学習する。つまりTCPで送られるこの4byteはプロトコル上完全なダミー値であり、プロキシが覗き見て学習すべき「動的ポート」は、TCPの中身ではなく、この後§3.3で述べる「サーバーがS→Cで送り返す4byte(サーバー割当UDPポート)」の方である。
- 加えて、hubは同一ポートで「JackTripWorkerが読む1発目」「JackTrip本体が読む2発目」の計2回、クライアントからの初回UDPパケット受信を経てからでないと本稼働(双方向音声パス確立)に入らない二段階構造を持つ(詳細な意図・クライアント送信頻度は送信側ループ実装(UdpDataProtocol.cpp、本調査未読了)に依存し未確認)。プロキシ実装がこの二段階タイミングをどう扱うかは追加調査事項として§8に追記する。
- 認証(`-A`/`mRequireAuth`)モードでは、TCP接続直後にまず平文4byteの往復(`Auth::OK` = `1<<16` = 65536 のLE整数)があり、双方が `Auth::OK` を確認した後にサーバーが `startServerEncryption()`・クライアントが `startClientEncryption()` を呼んでTLSハンドシェイクへ移行する。TLS確立後、クレデンシャル(username/password)とポート/クライアント名はすべてTLSレコードの内側(暗号化後)で送信される。認証応答コード(`Auth::OK`/`REQUIRED`/`NOTREQUIRED`/`WRONGCREDS`/`WRONGTIME`、いずれも65536以上の値)は同じ4byteフィールドをポート番号と共用しており、65535を超える値かどうかで両者を区別する設計。この認証フローがプロキシのTCP透過中継・パッシブ覗き見に与える影響の結論は §8 に記載する。

### 3.4 ヘッダの扱い(素通しする不変条件)

`DefaultHeaderStruct`(16byte)は次のフィールドで構成され、これらは**圧縮プロキシが一切変更せずそのまま転写する不変条件**とする。

- `TimeStamp`(uint64_t, 8byte)
- `SeqNumber`(uint16_t, 2byte)
- `BufferSize`(uint16_t, 2byte)
- `SamplingRate`(uint8_t, 1byte)
- `BitResolution`(uint8_t, 1byte)
- `NumIncomingChannelsFromNet`(uint8_t, 1byte)
- `NumOutgoingChannelsToNet`(uint8_t, 1byte)

理由は次の3点である。

1. `SeqNumber` はロス検出(受信側の `receivePacketRedundancy`)に使われており、プロキシが値を書き換えるとロス検出ロジックが誤動作する。
2. `BufferSize`・`BitResolution`・チャンネル数は受信側の `validatePeerHeader()` がペイロード長の逆算検証に使う自己記述フィールドであり、プロキシ自身もこれを読んでペイロード長を検算し、ペイロードの圧縮対象範囲を確定するために利用する。
3. `SamplingRate` は列挙値(uint8_t)であり、requirements.md §3.1 が明らかにした通り 384kHz は `UNDEF` として運ばれる。この値の意味を圧縮プロキシ側で解釈・変更する必要はなく、そのまま転写すればよい。

プロキシが手を加えるのはペイロード部(ヘッダに続く実サンプルデータ)のみである。

## 4. 実測データ

### 4.1 WavPack ベンチマーク結果

WavPack 5.9.0(CLI、Windows x64)を用い、10秒の合成音声素材3種(speech/music/noisefloor)× 2フォーマット(192kHz/24bit int、384kHz/32bit float)に対し、3条件(default / `-f` fast / `--blocksize=128`)でエンコードした。`--blocksize=128` は要件が想定する周期128サンプル相当の下限値である。

| 素材 | フォーマット | default 圧縮率 / 速度 | fast(-f) 圧縮率 / 速度 | blocksize=128 圧縮率 / 速度 |
|---|---|---|---|---|
| music | 192kHz/24bit | 78.25% / 115.32倍 | 78.71% / 154.69倍 | 89.63% / 27.31倍 |
| music | 384kHz/32bit float | 78.18% / 58.38倍 | 78.58% / 70.05倍 | 88.76% / 12.03倍 |
| speech | 192kHz/24bit | 38.29% / 141.66倍 | 38.41% / 187.33倍 | 48.35% / 44.47倍 |
| speech | 384kHz/32bit float | 42.95% / 83.27倍 | 43.04% / 98.55倍 | 51.50% / 20.62倍 |
| noisefloor(-80dBFS) | 192kHz/24bit | 46.03% / 125.6倍 | 46.00% / 160.0倍 | 57.51% / 28.34倍 |
| noisefloor(-80dBFS) | 384kHz/32bit float | 82.10% / 58.11倍 | 82.08% / 70.16倍 | 92.74% / 12.80倍(全条件中最悪) |

圧縮率は「圧縮後サイズ ÷ 圧縮前サイズ」であり、値が小さいほど帯域削減効果が大きい。速度倍率は10秒素材のエンコードに要した実時間に対する比(倍率が大きいほど高速)。

**限界注記**: 上記はいずれも Python + numpy で生成した合成素材(正弦波+一様乱数ノイズの近似)であり、実マイク収録音声ではない。実収録データ、特に超音波帯域まで伸びるマイク自己ノイズやアンプ由来の広帯域ノイズを含む素材では、本ベンチの noisefloor 条件(82〜93%)よりさらに圧縮率が悪化する可能性がある(詳細は §8)。またCLIによるファイル一括処理の計測であり、ストリーミングAPI(libwavpack 直叩き)での挙動とは異なりうる。

### 4.2 requirements.md §5.2「圧縮50%仮定」の検証結果

requirements.md §5.2 は「384kHz/32bit float × 圧縮ありで仮に50%削減」という仮置き値を用いていた。今回の実測では、**この仮定は素材依存であり、一律に成立するとは言えない**という結果になった。

- 50%を下回った(仮定より良い)のは、384kHz/32bit float の speech・default 条件(42.95%)の1点のみである。
- music(78.18〜88.76%)と noisefloor(82.10〜92.74%)は384kHz/32bit floatの全条件で50%を大きく上回り(圧縮率が悪く)、仮定より悪化した。
- リアルタイム用途に相当する `--blocksize=128` 設定では、384kHz/32bit float の speech(51.50%)も50%を上回り、この条件では素材を問わず50%仮定を満たさなかった。

したがって、「384kHz/32bit floatなら常に圧縮率50%を下回る」という requirements.md §5.2 の仮置き値は、speechかつdefault/fastブロックサイズという限定条件下でのみ成立する楽観値である。本文書は§5・§6でリアルタイム低遅延用途に相当する `--blocksize=128` 条件を実運用の代表値として採用しているため、その文脈に合わせるなら、実運用の平均としては約77.7%(3素材平均、§4.1表のblocksize=128列)、最悪ケース(noisefloor)では92.74%まで悪化しうる想定に置き換えるべきである。参考として、default/fast条件(非リアルタイム用途)の3素材平均は67.7〜67.9%程度である(数値の根拠は §4.1 表)。

### 4.3 プロキシ実コーデックによるストリーミング実測(2026-07-06、libwavpack 5.6.0 で検証)

§4.1 は wavpack.exe **CLI のファイル一括処理**の圧縮率であり、§8 で「プロキシが実際に行うパケット単位のストリーミング圧縮とは異なりうる」と留保していた。この留保を、**プロキシが実際に使う `gateway/codecs.py` のコーデック**(ctypes 経由の libwavpack ストリーミング API)で直接測定して解消した。ライブラリは Ubuntu 公式パッケージ libwavpack 5.6.0(`apt-get download` で取得)を使用し、全条件で送受信のビット一致(可逆性)を確認済み。合成素材である点は §4.1 と同じ限界を持つ。

**圧縮ブロックサイズ N(128サンプル/パケットを何パケットまとめて1回で圧縮するか)を変えたときの圧縮率(%)と、それに伴う片道バッチ遅延**:

192kHz/24bit(wavpack コーデック):

| 素材 | N=1 (0.7ms) | N=4 (2.7ms) | N=16 (10.7ms) | N=64 (42.7ms) |
|---|---|---|---|---|
| speech(会話近似) | 86.1% | 68.8% | **64.6%** | 66.5% |
| music(広帯域) | 106.3% | 86.4% | 82.0% | 81.0% |
| 無音+ノイズフロア | 67.6% | 51.4% | 47.3% | 46.3% |

384kHz/32bit float(wavpack コーデック):

| 素材 | N=1 (0.3ms) | N=4 (1.3ms) | N=16 (5.3ms) | N=64 (21.3ms) |
|---|---|---|---|---|
| speech | 97.3% | 83.3% | **79.9%** | 81.3% |
| music | 92.9% | 77.2% | 73.7% | 72.8% |
| 無音+ノイズフロア | 99.7% | 86.5% | 83.1% | 82.3% |

(zlib は全条件で wavpack より劣り、192kHz/24bit music では N によらず 98〜101%、384kHz/32float は 91〜97% とほぼ無圧縮。数値は `gateway` の zlib コーデックで別途測定)

**この実測から確定した重要事実**:

1. **N=1(パケット単位圧縮)は帯域削減にほぼ寄与しない**。128サンプル(768〜1024byte)はコーデックのブロックオーバーヘッドを償却できる大きさに達しておらず、music の int24 では **膨張(106.3%)** すら起きる。§5.1 が当初推奨した N=1 は、遅延最小の一方で圧縮効果が事実上ゼロであり、**帯域目的では不成立**。
2. 圧縮効果は **N=16 付近で頭打ち**(以降は逓減)。N=16 のバッチ遅延は 192kHz で片道 10.7ms、384kHz で 5.3ms と、150ms 予算に対し十分小さい。したがって圧縮を意味あるものにするには N≈16 を採るべきで、遅延コストは許容範囲。
3. 最良バッチでも **384kHz/32bit float は 73〜83%(17〜27%削減)止まり**であり、会話用途の 192kHz/24bit speech でも 65%(35%削減)程度。requirements.md §5.2 が仮置きした「50%削減」は達成できない。**圧縮単独では 384kHz×10人の帯域問題(§5.2)は解決せず、人数制限か回線増強との併用が必須**である。
4. なお統合テスト(§7.1)で観測した zlib 7.4% は dummy driver の**純粋なゼロ無音**という非現実的な最良ケースであり、実音声(ノイズフロアを持つ)の代表値は上表である。

## 5. 遅延予算

### 5.1 圧縮単位の選択

**当初 N=1(パケット単位圧縮)を推奨していたが、§4.3 のストリーミング実測により撤回する。** N=1 は追加遅延ゼロだが 128 サンプルではコーデックのブロックオーバーヘッドを償却できず、圧縮効果が事実上ゼロ(music では膨張)であることが実測で判明した。圧縮を帯域削減の手段として機能させるには複数パケットをまとめる必要がある。

**推奨: N≈16(圧縮効果の頭打ち点)**。§4.3 の実測で圧縮率は N=16 付近で頭打ちになり、それ以上まとめても改善は逓減する。N=16 のバッチ遅延は 192kHz で片道 10.7ms、384kHz で 5.3ms であり、150ms 予算(baseline 実測 ≈20ms)に対して十分小さい。したがって「N=1 で遅延最小・圧縮ゼロ」よりも「N≈16 で遅延を1桁 ms 増やして圧縮を実効化」する方が、帯域が律速となる本系統では合理的である。

**実装状況(2026-07-06)**: バッチ圧縮は `gateway.py` に実装済み(`--batch N` / `--batch-flush-ms M`)。同一フォーマットの音声パケットを N 個貯めて1回で圧縮し、フレームが N に達するか M ms 経過(ストリーム減速・停止時の遅延上限)で送出する。フレームはヘッダ群(zlib)と音声ペイロード群(選択コーデック)を分離して格納し、受信側はビット一致で個々のデータグラムに復元する。非音声(ハンドシェイク/制御)パケットとフォーマット変化時は現行どおり即時・単発送出し、順序を保つ。selftest でバッチ往復のビット一致(zlib/wavpack、N=2/16/64)を、実 JackTrip 統合テストで `--batch 16` の疎通(mismatches=0)を確認済み。既定は N=1(バッチ無効、後方互換)。

ただしトレードオフとして、まとめ圧縮は次を伴う:

- 送信側で N パケット分の圧縮フレームが揃うまで待つバッファリング遅延(下式)。`--batch-flush-ms`(既定 20ms)で、N に達しなくても部分バッチを一定時間で送出し遅延の上限を保証する。
- requirements.md §5.1 が前提とする `-q 8` のジッタバッファ設計(小さく規則的な遅延を許容する思想)に対し、N が大きいほどバースト的な到着になる。N=16 程度なら滞留時間(下記)と同オーダーで許容範囲だが、N=64 以上は避ける。
- 1圧縮フレーム内のパケットロスは N パケットまとめて失われる(ロスの粒度が粗くなる)。可逆圧縮フレームは部分デコードできないため、トンネル UDP のロス耐性設計(FEC/再送の要否)は段階2の検討事項とする。

複数パケットをまとめる場合(N≥2)の追加遅延は次式で見積もる。

```
追加遅延(片道、送信側バッファリング分) = (N - 1) × 周期時間
往復での実効増加 ≈ 2 × 追加遅延
```

- 192kHz/周期128の周期時間は `128 / 192000 ≈ 666.7µs`。N=4なら追加遅延 = 3 × 666.7µs ≈ 2.0ms(往復 ≈ 4.0ms)。N=2なら追加遅延 ≈ 666.7µs(往復 ≈ 1.33ms)。
- 384kHz/周期128の周期時間は `128 / 384000 ≈ 333.3µs`。N=4なら追加遅延 = 3 × 333.3µs ≈ 1.0ms(往復 ≈ 2.0ms)。N=2なら追加遅延 ≈ 333.3µs(往復 ≈ 0.67ms)。

`-q 8` のジッタバッファ滞留時間は `8 × 周期時間` で、192kHzでは `8 × 666.7µs ≈ 5.33ms`、384kHzでは `8 × 333.3µs ≈ 2.67ms` である。推奨の N=16 は 192kHz で往復約 21.3ms、384kHz で約 10.7ms を追加するが、150ms 予算に対しては十分小さい。N=1 は追加遅延ほぼゼロだが §4.3 のとおり圧縮効果がないため、帯域削減が目的なら採らない(圧縮を無効化して非圧縮運用する場合のみ N=1 に相当する)。

### 5.2 パケットレートと処理時間予算

1パケットあたりプロキシが処理を終えるべき時間予算は、次の周期時間と一致する(次のパケットが到着するまでに処理を終える必要があるため)。

- 192kHz/24bit/周期128/ch2: ペイロード = `128 × 2 × 3 = 768byte`、パケットサイズ = `16 + 768 = 784byte`、パケットレート = `192000 / 128 = 1500pkt/s`、処理時間予算 ≈ 666.7µs/パケット。
- 384kHz/32bit float/周期128/ch2: ペイロード = `128 × 2 × 4 = 1024byte`、パケットサイズ = `16 + 1024 = 1040byte`、パケットレート = `384000 / 128 = 3000pkt/s`、処理時間予算 ≈ 333.3µs/パケット。

WavPack実測(§4.1)の `--blocksize=128` 条件での速度倍率をこの処理時間予算に対する余裕として読み替えると、384kHz/32bit floatの最悪ケース(noisefloor、12.80倍)でも「10秒の音声を0.78秒で処理できる」ことに相当する。これを1パケットあたりの時間に換算する(1パケット時間 = 10秒 ÷ 速度倍率 ÷ パケット数、パケット数 = サンプルレート ÷ 128 × 10秒)と、384kHz noisefloor(12.80倍)で約26.0µs/パケット、他の条件でも約14.99〜27.71µsのオーダーとなり、1パケットあたりの処理は数十マイクロ秒オーダーで完了する計算になる(ナノ秒〜数マイクロ秒ではない)。それでも処理時間予算(384kHzで333.3µs、192kHzで666.7µs)に対しては一桁以上の余裕がある(ただしこれはファイル一括処理の平均値であり、パケット単位の実処理オーバーヘッド・スレッド切替コストは含まれていない点に注意。詳細は §8)。

### 5.3 requirements.md §5.1(片道150ms目標)への影響

hires-spike.md の実測では、192kHz/24bit・非圧縮PCMでの片道遅延は20ms未満(ネットワークRTT 4.4ms、JACK周期256/192000 ≈ 1.33ms、hubキュー`-q 8` ≈ 10.7msの合計)であり、150ms要件に対し一桁近い余裕がある。

圧縮プロキシをN=1(パケット単位)で導入した場合、プロキシ自体の処理時間(圧縮・伸長のCPU時間、往復で2回発生)が追加されるが、これは処理時間予算(192kHzで666.7µs、384kHzで333.3µs)の範囲内に収まる限り、既存の20ms未満という実測値に対して数百マイクロ秒〜1ミリ秒程度の上乗せに留まる見込みである。N=2でパケットをまとめる場合でも、往復増加は192kHzで最大1.33ms、384kHzで最大0.67ms程度であり、いずれも150ms要件に対して支障のない範囲と見込まれる。

ただし、この見積りはWavPackのファイル一括処理ベンチマークからの類推であり、パケット単位のストリーミング実処理での実測ではない。実装後の実測による確認が必須である(§7 PoC計画)。

## 6. CPU/RAM 試算

requirements.md §5.4 はStage 2でVPSをRAM 2GBクラスへ引き上げることをSHOULDとしている。この前提のもと、2vCPU級VPSでのCPU負荷を試算する。

WavPack実測(§4.1)の `--blocksize=128` 条件での速度倍率を「1/速度倍率 × 100%」でCPU占有率相当に換算すると、次のようになる。**この換算式は「WavPack CLIがシングルスレッドで動作している」ことを前提とした未検証の仮定に立つ**(§8で述べる通り、CLIが内部でマルチスレッド処理をしている可能性があり、`--no-threads`指定時の速度は未計測)。もしCLIが複数コアを使って58.11倍・12.80倍等の速度を達成しているなら、「1/速度倍率」は複数コア分の合成能力を1コアの専有率であるかのように扱うことになり、以下の値は実際のシングルコア専有率を過小評価している可能性がある。

- music: 192kHz ≈ 3.7%、384kHz ≈ 8.3%
- speech: 192kHz ≈ 2.2%、384kHz ≈ 4.8%
- noisefloor(最悪ケース): 192kHz ≈ 3.5%、384kHz ≈ 7.8%

2vCPU(合計CPUキャパシティ200%と仮定)で、最悪ケース(noisefloor、384kHz、7.8%/ストリーム)の同種ストリームを10本同時に処理する場合の概算は次の通り。

```
7.8% × 10ストリーム = 78% ≪ 200%
```

この単純積算だけを見ればエンコード処理には余裕があるように見えるが、以下の留保を踏まえると結論はより不確実である。

- WavPack CLIの計測はエンコード処理のみであり、ヘッダ検算・ネットワークI/O・スレッド管理を含むプロキシ全体のオーバーヘッドは含まれない。
- ローカルWindows機での計測結果からの外挿であり、VPS実機(CPUアーキテクチャ・クロック速度・仮想化オーバーヘッドが異なりうる)での再検証が必要。
- 10ストリーム同時稼働時にはエンコードとデコードの両方が発生する(hub側は全クライアント分のエンコード/デコードを担う)ため、実際の負荷は上記の単純積算よりも高くなる可能性がある。仮にデコードもエンコードと同等の負荷(7.8%/ストリーム)と見積もり単純に2倍すると、`7.8% × 2 × 10 = 156%` となり、200%に対する余裕は78%との単純積算より大幅に小さくなる。
- hub の既存負荷(requirements.md §5.3実測: 2クライアント時でjacktrip ≈28% + jackd ≈6%、1コア比)にプロキシの負荷が上乗せされる点も考慮が必要。この既存負荷を1クライアントあたり `(28+6)/2 = 17%` とみなして10クライアントへ単純に線形外挿すると `17% × 10 = 170%` に相当し、これ単独でも200%の大部分を占める規模になる。

エンコード+デコードの156%相当と、既存hub負荷の線形外挿170%相当を踏まえると、これらは重複してカウントすべきではない部分がある一方(既存実測にはある程度JackTrip自体の処理が含まれる)、いずれの試算も単純な「78% ≪ 200%」という余裕の主張とは整合しない規模である。したがって、**「理論上は10ストリーム分の余裕がある」と断定することはできず、現時点のデータからは10ストリーム同時運用時のCPU余裕は不確実というのが正しい結論である**。default設定(58.11倍速、1.7%/ストリーム相当)であればエンコードのみで10ストリームでも17%程度に収まるが、これも同様にデコード・既存負荷を考慮していない点に留意する必要がある。実際の可否はVPS実機での多クライアント実測(§7 PoC計画)を待つ必要がある。

RAM消費については、既存実測(2クライアントでオーディオスタック合計 ≈400MB、requirements.md §5.3)にプロキシプロセスのメモリ使用量(ポート対応表・圧縮バッファ等)が追加されるが、WavPackの圧縮処理自体は小ブロック(128サンプル)を都度処理する設計であるため、大きなメモリ増加は見込まれない。ただし具体的な増加量は実装・実測が必要であり、本文書の時点では数値を確定できない。

## 7. PoC 計画

requirements.md §7 Stage 2 exit criteria「圧縮ゲートウェイが192kHz/24bitで片道150ms以内・実測圧縮率を計測済み」の達成に向け、2段階でPoCを実施する。

### 7.1 段階1: ローカル(WSL/同一マシン内)

- **構成**: 同一マシン上(またはWSL⇔Windowsホスト間)にJackTripクライアント・hub・両側プロキシをすべて配置し、ネットワーク遅延・帯域制約を排除した状態でプロキシ自体の正しさを検証する。
- **検証項目**:
  1. TCP 4464ハンドシェイクの透過中継が既存のTLS ClientHello判定ロジック(平文ポート送出時の先頭バイトパターン)と衝突しないこと。
  2. VPS側プロキシがTCPストリームからサーバー割当UDPポート(hub→client 方向の4byte little-endian整数。§3.3 の訂正どおり、client→hub 方向の4byteはダミーで学習対象ではない)を正しく学習し、対応するUDPポートペアへの1:1中継が確立すること。
  3. ヘッダ(16byte)が一切変更されず素通しされていること(TimeStamp/SeqNumberの値をキャプチャして送受で一致を確認)。
  4. ペイロード部の圧縮/伸長が可逆であること(送信前と受信後のPCMサンプル値がビット一致すること)。
  5. 192kHz/24bit・周期128でのJackTrip接続がプロキシ経由でも成立し、XRUNが発生しないこと。
- **成功基準**: 上記5項目すべてがPASSし、かつ192kHz/24bitでの実測圧縮率とプロキシ導入による追加遅延(ループバック測定)を数値で得ること。requirements.md §7 Stage 2 exit criteriaの「JackTripのUDPペイロード構造と圧縮データの互換性を検証済み」を、本設計文書の結論(案B採用)に沿って実装レベルで裏付ける段階と位置づける。

**進捗(2026-07-06)**: `gateway/` 配下にプロトタイプを実装した(Python asyncio、コーデックは none/zlib/wavpack のプラガブル構成)。

- `gateway/codecs.py`(558行): コーデック層。`wavpack` は ctypes による libwavpack ストリーミングAPI(`WavpackPackSamples`/`WavpackUnpackSamples`)バインディング。
- `gateway/gateway.py`(784行): 透過トンネルプロキシ本体(`--role hub/client`、TCP handshake sniffer、UDP中継、トンネルフレーミング、統計、`--debug-headers`)。
- `gateway/selftest.py`(358行): ネットワーク不要のユニットセルフテスト。
- `gateway/POC.md`(182行): 段階1(ローカル/WSL)実行手順書。

selftest 実行結果(`python3 gateway/selftest.py`、libwavpack 5.6.0 を `GATEWAY_WAVPACK_LIB` で指定): **`PASS=24 FAIL=0 SKIP=0`**。内訳は codec:zlib と **codec:wavpack** の int24/float32 往復(random/sine/silence、各1000回、**全ビット一致**)、ハンドシェイクスニファ(1回読み/分割読み/`Auth::OK`(65536)値の誤検出防止)、トンネルフレーム往復(4種のcodec/tag/len組合せ+不正マジック/切詰めヘッダの拒否)、ヘッダ素通し(codec=none/zlib、およびサブヘッダ長データグラムの強制パススルー)がすべてPASS。ライブラリ未指定時は wavpack のみ SKIP(FAILにはしない)。

wavpackコーデックは当初 ABI レベルの静的検証のみだったが、2026-07-06 に Ubuntu 公式パッケージ **libwavpack 5.6.0**(`apt-get download` で取得、sudo 不要)を用いて **実機ラウンドトリップ(全条件ビット一致)を確認済み**。ctypes バインディングの `WavpackConfig` フィールド順序・`total_samples=-1`+`QMODE_IGNORE_LENGTH`・`OPEN_STREAMING` 等が実ライブラリで正しく機能することを実証した。プロキシ実コーデックによるストリーミング圧縮率の実測は §4.3 を参照。

**統合テスト実施(2026-07-06、WSL Ubuntu・jacktrip 2.2.2/jackd 1.9.21・codec=zlib)**: 実 JackTrip hub + 両ゲートウェイ + 実 JackTrip クライアントを、単一マシン上で IP 分離(hub 側=127.0.0.1、client 側=127.0.0.2)して接続する統合テストを実施した。この過程で **client 役ゲートウェイの実バグを1件発見・修正した**。

- **発見したバグ**: `_register_client_session()` がローカル UDP をエフェメラルポート(`listen_port=0`)にバインドしていた。しかし JackTrip クライアントは「hub が通知したポート(通知値)宛」に音声を送る(クライアントログ `Server Address set to: 127.0.0.2 Port: 5466` で確認)ため、エフェメラルポートでは音声を捕捉できず、UDP がゲートウェイを一切通らなかった(統計行が出力されず)。
- **修正**: client 役はローカル UDP を「hub 通知ポートそのもの」にバインドするよう変更(`listen_port=hub_udp_port`)。修正後、統計行 `[stats] pkts=8139 in=4297392B out=317421B ratio=7.4% mismatches=0` が出力され、実 JackTrip クライアントの音声パケットがゲートウェイを通過するようになった。selftest は修正後も 18 PASS を維持。

| 2 | 両ゲートウェイによる割当UDPポート学習と session tag 一致 | **PASS(実 JackTrip)**。hub 役・client 役の両ゲートウェイがハンドシェイクから同一の hub 割当ポート(5466)を学習し、`session tag 5466` で一致(`learned hub udp port 5466 -> session tag 5466` が両ログに出力) |
| 3 | ヘッダ(16byte)の素通し・検算 | **PASS(実 JackTrip)**。実 JackTrip 音声パケット 8000 発超を中継し、ヘッダ自己記述検算の不一致(`mismatches`)が 0 のまま。ヘッダを壊さず正しくペイロード長を算出できていることを実データで確認 |
| 4 | ペイロード圧縮/伸長の可逆性 | **PASS(zlib、実 JackTrip 音声で往復)**。実クライアントの音声を zlib で往復させ検算不一致 0。圧縮率は無音(dummy driver)のため 7.4% と参考値。wavpack は実ライブラリ不在で未検証(ユニットレベルは ABI 静的検証のみ) |
| 5 | 双方向接続確立・XRUN非発生の持続 | **未達(単一マシンの制約、設計欠陥ではない)**。client→hub 方向は音声が流れたが、逆方向(hub→client)が確立せずクライアントは `Waiting for Peer...` のまま。原因は、実 hub の JackTripWorker が UDP をデュアルスタックのワイルドカード(`[::]:5466`)でバインドするため、client 役ゲートウェイの同ポートバインドと衝突し `Could not bind UDP socket` で hub 側ワーカーが起動できないこと。`-L` による IP 指定でも回避できない。**実運用では client と hub が別ホストのため衝突しない**。単一マシンでの完全な双方向検証には netns 分離(root 必要、当環境は sudo パスワード制のため不可)か2ホスト構成が要る。実質的にこれは段階2(VPS、2ホスト)で自然に検証される |

**総括**: 最もリスクの高い「実 JackTrip のハンドシェイクをバイト解析してポートを学習し、実音声パケットのヘッダを壊さず検算してペイロードだけ圧縮する」経路は、実 JackTrip 相手に end-to-end で動作を確認できた(検証項目1〜4）。加えて統合テストによって client 役の UDP バインド漏れという実バグを発見・修正できたことは、PoC 段階1の主要な収穫である。残る双方向確立(項目5)は単一マシンのポート衝突という環境制約が原因であり、段階2(VPS 2ホスト)で検証する。

### 7.2 段階2: VPS

- **構成**: 段階1で動作確認したプロキシをVPS側・クライアント側それぞれに実配置し、実際のWAN経路(クライアントPC⇔VPS)で検証する。
- **検証項目**:
  1. 実ネットワーク経路(hires-spike.mdの実測RTT 4.4ms相当)でのプロキシ経由接続が成立し、安定して維持されること。
  2. 片道遅延の実測(requirements.md §5.1の手順に準拠したループバック録音)が150ms以内に収まること。
  3. 実測圧縮率(合成素材ではなく、実際の音声入力または既存の192kHz/24bit録音済み音声を用いた再生入力での計測)。
  4. 2クライアント同時接続時のVPS側CPU/RAM実測(requirements.md §5.3の既存実測400MB・jacktrip 28%+jackd 6%との比較でプロキシの上乗せ分を切り分ける)。
  5. gClientGoneTimeoutMs(10秒)以内にプロキシ側の異常(再起動・一時的なフリーズ)から復旧できるかの耐障害性確認。
- **成功基準**: 192kHz/24bitにおいて、片道150ms以内・実測圧縮率・VPS実機でのCPU/RAM実測値がすべて得られ、requirements.md §7 Stage 2 exit criteriaの残り2項目(「VPSを2GBクラスに引き上げ、RAM実測が許容範囲内」「圧縮ゲートウェイが192kHz/24bitで片道150ms以内・実測圧縮率を計測済み」)を満たすこと。

## 8. 未解決点

以下は本設計文書の時点では実装を伴わないと判明しない事項、またはベンチマークの限界として残るものである。断定はせず、実装・追加検証が必要な点として明記する。

- **実マイク素材での圧縮率**: 本ベンチは合成素材(正弦波+一様乱数ノイズ)によるものであり、実マイク収録(超音波帯域まで伸びるマイク自己ノイズ・アンプ由来の広帯域ノイズを含む)では、今回のnoisefloor素材(82〜93%)よりもさらに圧縮率が悪化する可能性がある。定量値は実収録データでの追試が必要。
- ~~libwavpackストリーミングAPIとの差~~(**解消済み 2026-07-06、§4.3**): プロキシ実コーデック(libwavpack 5.6.0 ストリーミング API)でパケット単位・バッチ単位の圧縮率を直接実測した。CLI 一括処理(§4.1、48〜93%)との差は予想通り大きく、N=1 では 67〜106%(ほぼ無圧縮〜膨張)、N=16 でようやく 47〜82% に到達する。これを受けて §5.1 の圧縮単位推奨を N=1 → N≈16 に修正した。
- **シングルスレッド動作時の速度**: CLIが内部でマルチスレッド処理をしている可能性があり、`--no-threads`指定時の速度は未計測。実際のプロキシ実装がシングルスレッドで動作する想定の場合、CPU占有率換算(§6)が本測定より悲観的になる可能性がある。
- **VPS実機でのCPU/RAM**: §6の試算はローカルWindows機の計測結果からの外挿であり、VPS実機(2vCPU、CPUアーキテクチャ・クロック速度・仮想化オーバーヘッドが異なりうる)での実測が必要。
- **TCPハンドシェイク覗き見の頑健性**: プロキシがTCP 4464ハンドシェイクを覗く実装は、TLS(証明書設定時)・認証(`mRequireAuth`)フローの可変長フィールドとバイト境界で衝突しないよう、JackTrip側の最新のハンドシェイクロジックに追随させる必要があり、JackTripのマイナーバージョンアップで仕様が変わると学習ロジックが壊れる可能性がある。
- **認証(`-A`)モード時のパッシブ覗き見は不可能、MITM構成が必須(2026-07-06 ソース調査で確定)**: JackTrip mainブランチのソース(UdpHubListener.cpp/JackTrip.cpp/Auth.h)を読了した結果、`-A`(`mRequireAuth`)有効時は、TCP接続直後の平文4byte往復(`Auth::OK`確認)の後、`startServerEncryption()`/`startClientEncryption()` によりTLSハンドシェイクへ移行し、以降のusername/password・クライアント名・ポート値はすべてTLSレコードの内側(暗号化後)でのみ送受信されることが確定した。透過型(パケットをそのまま右から左に転送するだけ)のプロキシはTLSレコードの中身を一切復号できないため、ユーザー名・パスワードは無論、認証の成功/失敗すら判別できない。**結論(verdict): 認証モードでのパッシブな覗き見は不可能。** 対応方針として、`-A` 構成をサポートする場合は、VPS側プロキシが外部からの接続を一旦受け、hubと同一の証明書・鍵(想定: `/etc/selfmatrix-hires/tls.{crt,key}` 等の共有配置)でTLSを終端し、復号した内容を検査・ポート学習した上で改めてhubへ(別のTLS接続または平文で)中継するMITM構成へと設計変更する必要がある。これは案B(透過トンネル、TCPバイト単位の完全パススルー)からの逸脱であり、追加の実装コスト・証明書運用(hubとプロキシ間での鍵共有)・信頼境界の見直しを伴う。**現時点のPoCスコープ(§7)は認証なし構成のみを対象とし、認証あり構成のMITM実装は本設計文書のスコープ外の将来課題とする。** なお平文モード(認証なし)では、TCPで送られる4byte自体はhub側で実UDP宛先ポートとして採用されないダミー値であり(§3.3参照)、UDP音声パスは認証の有無によらず常に平文であるため、UDP側の観測・中継は認証設定と無関係にパッシブに可能である。
- **二段階UDP初回パケット受信の扱い**: hubは同一ポートで「JackTripWorkerが読む1発目」「JackTripが読む2発目」の計2回、クライアントからの初回UDPパケットを受信してから双方向音声パスを確立する(§3.3参照)。この二段階構造がクライアント側の送信ループ実装(UdpDataProtocol.cpp、本調査で未読了)とどう噛み合うか、プロキシがこのタイミングをどう中継すべきかは追加調査が必要。
- **ポート対応表の競合状態**: 複数クライアントが同時に接続・再接続を繰り返す場合の、プロキシ側ポート対応表の管理方式や、クライアントごとの対応表構築時の競合状態(race condition)への対処は運用設計次第であり、本文書だけでは確定できない。
- **`port_tag = 割当ポート & 0xFFFF` の折り返し前提**: 統合テスト(§7.1、hub を `-B 14464` で起動)で、hub の UDP base port が 71002(`-B` のオフセット 10000 が base port にも連動加算される JackTrip 2.2.2 の未文書化挙動)となり、割当ポートが 65535 を超えて `71002 & 0xFFFF = 5466` と折り返った。単一セッションでは両プロキシが同じ折り返し値を導くため無害だったが、`gateway.py` docstring の「`mBasePort <= 64511` なので truncation は起きない」という前提は運用者の `-B`/`-U` 設定次第で崩れる。複数セッションで異なる id が同じ下位16bitに衝突する縮退ケースがあり得るため、段階2では port_tag を割当ポートの全32bitで持つか、hub の base port を 65535 未満に収める運用制約を課すべき。
- **RT優先度設定の可否**: プロキシ側スレッドに`--udprt`相当のSCHED_FIFO優先度やソケットQoS(DSCP EF)を設定する場合、OS側のリアルタイムスケジューリング権限(Linuxではrtprio capabilityやcgroup設定)が必要になり、VPS環境(コンテナ/仮想化)によっては付与できない制約がある可能性がある。
- **プロキシ再起動時のセッション扱い**: プロキシ再起動をgClientGoneTimeoutMs(10秒)以内に収めればJackTrip側セッションは維持されるが、学習済みのポート対応表が再起動をまたいで失われる場合、新規接続のポート学習ができない期間が生じる。永続化やホットリロード機構の要否は実装時に判断する必要がある。
- **JackTripライセンス**: requirements.md §6/§9で既に未解決課題とされている通り、JackTrip本体のライセンス(AGPL-3.0系と見られる)についての正式確認は本文書でも解決していない。案Bはプロキシを別プロセスとして実装するためライセンス条件と抵触しない見込みだが、これは既存の留保付き見込みの域を出ない。
- **セッションタイムアウト定数の未確定点**: §3.3 で参照している `gClientGoneTimeoutMs = 10000` は、実際にこの定数を参照してタイムアウト判定を行っているコード(発火ロジック)が本調査対象ファイル(JackTripWorker.cpp/JackTrip.cpp)内で特定できず、UdpDataProtocol.cpp 等の未読了ファイルにある可能性が高い(値の定義自体は jacktrip_globals.h で確認済み)。また `gTimeOutMultiThreadedServer = 10000` はコメントが `// seconds` だが使用箇所(100ms 刻みの積算比較)からはミリ秒として運用されているとしか読めず、コメントと実装で単位表記が食い違っている(JackTrip 側のコメントミスの可能性)。いずれも「10秒」という前提でプロキシ設計を進めてよいと見込むが、正確な適用ロジックと意図した設計値は未確認であり、プロキシの再起動許容時間を厳密に詰める段階では実測(実際に何秒でセッションが切れるか)で裏取りする必要がある。

## 9. requirements.md への影響(更新はしない。リストのみ)

本設計文書の結論を受けて、requirements.md の以下の箇所は将来的な更新が必要と考えられる。ただし本文書では更新を行わず、対応すべき箇所の指摘に留める。

- **§4.1「実現方式」の記述**: 「挟む」の内実を本文書の結論(案B: JackTrip無改造+外側UDPプロキシによる透過トンネル圧縮)に合わせて具体化し、ペイロード直接挟み込み(案A、不採用)ではなく透過トンネル型プロキシ(案B、採用)であると明確化する。
- **§5.2「圧縮50%仮定」**: 「384kHz/32bit float(圧縮あり、仮に50%削減)」という仮置き値を、本文書§4.2の実測(素材依存で50%を上回る場合が多い)を踏まえ、素材依存の注記、またはリアルタイム用途(blocksize=128)を基準とした約77.7%(平均)・92.74%(最悪ケース)への数値見直しを行う(非リアルタイムのdefault/fast条件では67.7〜67.9%程度)。
- **§7 Stage 2 exit criteria**: 「JackTripのUDPペイロード構造と圧縮データの互換性を検証済み」の項目に、本文書の設計判断(案B採用)と、残る実装・実測(§7 PoC計画の段階1・段階2)完了をもって満たされるという段階分けを反映する。
- **§9 未解決課題「圧縮ゲートウェイの実現可能性」**: 「両立しない場合、別トランスポートの自作が必要」という分岐を「案Bで成立する」という結論に更新し、§8の未解決点(実マイク素材、ストリーミングAPI差異、VPS実機CPU/RAM等)を新規未解決課題として追加する。
- **§6「制約と既知の事実」**: WavPackの実測圧縮率・CPU負荷が「一次情報から得られていない」という記述を、本文書§4.1の実測データへの参照に更新する。
