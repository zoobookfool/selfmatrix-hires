# Requirements — selfmatrix-hires

2026-07-06 のゼロベース再定義です。設計・実装の判断はこのドキュメントを正とします。

## 0. 位置づけ

selfmatrix-hires は [SelfMatrix](https://github.com/zoobookfool/selfmatrix) 本体とは結合しない独立拡張です(2026-07-05 決定)。本ドキュメントは **このリポジトリの要件の正本** であり、親プロジェクトの `docs/requirements.md` §4(ハイレゾ音声)および §9 の該当決定事項は、本ドキュメントへの参照に差し替えます(§11 参照)。親プロジェクト側で本ドキュメントの内容を転記・重複維持する必要はありません。

分離の動機(2026-07-06 決定): 本体分離によって「本体の最小スコープを乱さない」という制約が外れたため、ハイレゾ系統は独立オプションとして目的からゼロベースで見直し、やりたいことを一通り要件に載せる。ただし「載せる」ことと「今すぐ作る」ことは別であり、段階的ロードマップ(§7)で優先順位を裁く。

本系統は、本体の通話 E2EE 方針(親 `requirements.md` §3 / `architecture.md`)の適用対象外であり、別系統として意図的に平文伝送を許容する(詳細・理由は §4.2)。

タグの意味(親プロジェクトと同一の体系):

- `MUST`: 初期リリース(Stage 1、§7)までに満たす
- `SHOULD`: 満たしたいが、実装コスト次第で時期を調整できる
- `LATER`: 将来対応。今は設計で道を塞がないことだけ担保する
- `OUT`: 当面対象外と明示的に決めたもの

各要件には根拠(実測値・調査事実)と検証方法を付す。検証方法が書けない要件は書かない。

## 1. 目的とユースケース

### 1.1 選択したユースケース

- **会話の高音質化**: 通話音声そのものの品質を、Discord/Opus 相当を超えるレベルに引き上げる `MUST`
  - 検証方法: 聴感 ABX テスト(48kHz/Opus vs 192kHz/24bit PCM)で有意差が確認できること(§9 未解決課題)
- **収録・録音素材の取得**: セッションの音声を、配信・アーカイブ・編集に使える素材として記録する `MUST`
  - 検証方法: 録音方式(§4.3)で定めた形式のファイルが、セッション後に破損なく取り出せること

### 1.2 明示的に選ばなかったユースケース(OUT、理由付き)

- **音楽鑑賞・リスニング会** `OUT`
  - 理由: ユーザー決定によりスコープ外。BGM 再生・音源の高音質再生を主目的とした機能(共有再生・同期プレイヤー等)は設計しない
- **楽器セッション・合奏** `OUT`
  - 理由: ユーザー決定によりスコープ外に加え、技術的にも別要件領域。合奏は「音楽的に成立する遅延」(一般に片道 20〜30ms 以下が目安とされる)が必要になり、本要件の遅延目標(§5.1、片道 150ms 以内)より一桁厳しい。同じ系統で両立させようとすると遅延要件が過剰に厳しくなり、会話用途の帯域・機材ハードルとトレードオフが生じるため、目的を分離したまま合奏専用の要件を別途起こす方が健全と判断する

## 2. 全体方針

- 送信側は専用ネイティブアプリ + 対応オーディオインターフェース前提とする `MUST`
  - 根拠: ブラウザのマイク取得は実質 48kHz 止まりであり、ハイレゾ伝送の入口になり得ない
- 参加者はヘッドセット必須(運用ルール) `MUST`
  - 根拠: JackTrip hub にはエコーキャンセル機能がなく、スピーカー参加はハウリング・エコーの原因になる(現行 README §5.1 記載の既知事実)
- 本体(クライアント fork)との非結合を維持する `MUST`
  - 起動導線・自動ミュートは本体に実装しない。二重再生防止は運用ルールで担保する(§6.2)
- 映像とのリップシンク保証 `OUT`(当面)。音声主体の設計とする

## 3. 音質段階と対応目標

### 3.1 段階定義

| 段階 | サンプルレート/ビット深度 | 位置づけ |
|---|---|---|
| Stage 0(現状) | 192kHz/24bit、非圧縮 PCM | 実装済み(現行 README/provision.sh) |
| Stage 1 目標 | 192kHz/24bit を安定運用 | 本要件の初期リリース(半自動ランチャーは含まない。ランチャーは Stage 2、§4.4/§7) |
| 対応目標(上限) | **384kHz/32bit float** | 段階的に引き上げる最終目標(本ドキュメントで正面目標に格上げ、旧要件は 192kHz/24bit を SHOULD・384kHz/32bit を LATER としていた) |

- 384kHz/32bit float を対応目標とする `SHOULD`(Stage 3、§7)。ただし実機検証未了のため `MUST` にはしない
  - 根拠(技術的整合性): JackTrip の `-b 32` は int32 へのビット拡張ではなく、JACK 内部表現である `sample_t = float`(32bit IEEE754)を `memcpy` でそのまま伝送する「本物の float32」であることをソースコード(`AudioInterface.cpp`)で確認済み。384kHz/32bit float という組み合わせは JackTrip の実装上は矛盾しない
  - 根拠(列挙型調査の確定結果、2026-07-06 ソース直接確認): `AudioInterface.h` のサンプルレート列挙型 `samplingRateT` は `SR22〜SR192 + UNDEF` で **384kHz の列挙子は存在しない**(v2.2.2 / v3.0.0 / main で完全同一)。384000Hz は `getSampleRateType()` で必ず `UNDEF` に丸められる。ただしパケットヘッダ(`DefaultHeader`)はサンプルレートを生 Hz 値ではなく列挙値(uint8_t)で運び、ピア検証(`checkPeerSettings`)は**列挙値の一致比較のみ**を行うため、hub・クライアント双方を 384kHz に揃えれば両者とも `UNDEF` で「一致」し接続は成立する見込み。音声データパス(JACK バックエンド)にはレート上限チェックが存在しない
  - **実証的裏付け(重要)**: `getSampleRateType()` の SR192 判定には `19200`(正しくは `192000`)というタイポが存在し(`AudioInterface.cpp:799`、main で確認済み)、**現行の 192kHz 運用も実はヘッダ上 `UNDEF` 扱いで動作している**。2026-07-05 のスパイク PASS はこの UNDEF モードでの実績であり、UNDEF モード自体が実用に耐えることは既に実証されている。384kHz は現行 192kHz と同じコードパスを通ることになる
  - リスク: UNDEF モードではサンプルレート不一致検出が機能しない(異なる非対応レート同士でも `UNDEF` 一致で接続が成立し、ピッチずれとして顕在化する)。ログにも `Sampling Rate = 0` と誤表示される。**全参加者のレート一致は半自動ランチャーの配布設定(§4.4)と運用ルールで担保する必要がある**
  - 正式対応(`SR384` 列挙子追加 + `19200` タイポ修正)にはアップストリーム改修またはフォークパッチが必要。列挙子を末尾に追記する形であればワイヤーフォーマット(uint8_t 1バイト)は不変で既存レートとの互換性は維持される
  - 検証方法: Stage 3 では UNDEF モード前提で、hub・クライアント双方を 384000Hz で起動し、jackd/JackTrip がエラーなく起動し XRUN なく接続が成立することを実機(dummy driver ではなく実オーディオデバイス)で確認する。実際に 384kHz のまま伝送されていることは**実測帯域**(1人あたり片方向 ≈24.6Mbps、§5.2)で確認する(ヘッダの `SamplingRate` フィールドは `UNDEF` 固定のため確認手段にならない)。あわせて `checkSampleRate()`/`getClosestSampleRate()`(RtAudioバックエンド、§6)による黙示的なリサンプリングで見かけ上エラーなく動いているだけの状態と区別する。Stage 3 の成果物として、上流への `SR384` 追加 + `19200` タイポ修正パッチの contribution(またはフォークパッチ運用)の要否判断を含める

### 3.2 機材面の制約(384kHz の意味を正しく裁く)

- 384kHz/32bit float に対応する市販オーディオインターフェースは、RME ADI-2 Pro シリーズ等の実売 $1,500〜2,000 級のハイエンド機にほぼ限られる。192kHz 止まりの製品(RME Babyface Pro FS、Focusrite 2i2/4i4、Zoom UAC-232 等)が民生・プロシューマー市場の主流であり、384kHz 化は参加可能な機材保有者を大幅に絞る変更である `既知の制約として明記`
- 一般的なヘッドセット/コンデンサーマイクの周波数特性は 20Hz〜20kHz 程度が標準で、384kHz サンプリング(理論上 192kHz までの周波数成分を記録可能)を与えても、マイク側が可聴域を超える情報を収音できないため「サンプル数は増えるが新規の情報量は増えない」点に留意する `既知の制約として明記`
  - この制約は Stage 3 を無意味にするものではない(ジッター耐性やアンチエイリアシングフィルタの位相特性改善など、録音チェーン全体の品質向上には寄与し得る)が、「384kHz にすれば聴こえが劇的に変わる」という前提は置かない
- 上記よりStage 3(384kHz/32bit float)は「対応可能な参加者は使える上位互換オプション」として設計し、非対応の参加者は Stage 1/2 相当(192kHz/24bit)にフォールバックできる構成とする `SHOULD`
  - 検証方法: サンプルレート不一致時のフォールバック手順(hub 建て直し)が現行 README §2 の 192kHz/48kHz 切替と同じ運用パターンで回せること

## 4. 機能要件

### 4.1 伝送

- 非圧縮 PCM 伝送(現行方式)を Stage 1 の基盤として維持する `MUST`
  - 根拠: JackTrip 実装は非圧縮 PCM 固定であり、追加開発なしで動く現実的な選択肢
- 可逆圧縮伝送を追加する `SHOULD`(Stage 2、§7)
  - 候補比較: WavPack は 32bit float PCM をネイティブ・可逆に扱え、OSS ライセンス(BSD 類似)で再配布制約が緩く、低遅延ブロックサイズでの先行研究(Kurtisi et al., ICME 2008、ブロックサイズ 128 サンプル runs)がある。FLAC は仕様上「整数のみ」(RFC 9639)で float32 非対応のため、使うなら float→int32/24 変換を前段に挟む設計が必要になる。以上より **WavPack を第一候補とする**
  - ライセンス(要確認): JackTrip 本体のライセンスは AGPL-3.0 系と見られる(§6/§9 で要再確認)。圧縮ゲートウェイを JackTrip にリンクせず別プロセス(前後に挟む独立プロセス)として実装する設計であればライセンス条件と抵触しない見込みだが、正式な確認は未了のため §9 の未解決課題として扱う
  - 圧縮率の前提(実測済み 2026-07-06、合成素材): WavPack 5.9.0 の blocksize=128(周期128相当)実測で speech 48〜52%、music ≈89%、ノイズフロア 384kHz/32bit float 92.7%(最悪)。**小ブロックでは圧縮効果が素材依存で大きく振れるため、「圧縮すれば帯域問題が解決する」という前提は置かない**([docs/stage2-compression-gateway.md](stage2-compression-gateway.md) §4)。実マイク素材(超音波帯ノイズフロアを含む)ではさらに悪化しうる
  - 実現方式(**設計確定 2026-07-06**、[docs/stage2-compression-gateway.md](stage2-compression-gateway.md)): 案A(JackTrip の UDP ペイロードに直接圧縮データを挟む)は、受信側 `validatePeerHeader()` の自己記述的サイズ検証(ヘッダから期待パケット長を逆算)が可変長ペイロードと衝突するため**不成立**。**案B: JackTrip 無改造 + 外側 UDP プロキシによる透過トンネル圧縮**(ヘッダ 16byte は素通し、ペイロードのみ圧縮。TCP 4464 ハンドシェイクをパッシブに覗き見てサーバー割当 UDP ポートを動的学習)を採用する。圧縮単位は N=1(パケット単位、追加バッファリング遅延ほぼゼロ)を推奨。実装・実測(PoC 段階1/2)は未了
  - 検証方法: 設計文書 §7 の 2 段階 PoC。段階1(ローカル)でハンドシェイク透過中継・ヘッダ素通し・圧縮の可逆性(ビット一致)・接続成立を、段階2(VPS)で実 WAN 経路の片道 150ms 以内(§5.1)・実測圧縮率・VPS 実機 CPU/RAM を確認する
- 非圧縮 PCM のみでの運用(圧縮なし)は帯域試算(§7)を満たす人数・回線の範囲内で `LATER` ではなく継続利用可能とする(Stage 1 はこの状態)

### 4.2 認証・セキュリティ

- hub 接続には認証を必須とする `MUST`(現行実装済み: `-A` + credsfile)
  - 根拠: ソースコード調査により、JackTrip の TCP ハンドシェイクには TLS 認証(`-A`/`--certfile`/`--keyfile`/`--credsfile`)が存在することを確認済み
- **UDP 音声ペイロードは暗号化されない**ことを既知の制約として明記する `既知の制約として明記`
  - 根拠: `UdpDataProtocol.cpp` を直接確認し、暗号化・TLS/DTLS 処理が音声送受信経路に一切実装されていないことを確認した。`-A` による認証は接続確立時の TCP ハンドシェイクのみを保護し、音声を運ぶ UDP ストリーム自体は平文で流れる
  - 対応方針: 盗聴・改ざん耐性が必要な運用(公衆網を跨ぐ等)では、WireGuard 等の VPN や SSH トンネルの併用を運用者の判断に委ねる。ただし JackTrip 公式は VPN 経由の利用を高遅延化の原因として非推奨としており、暗号化と低遅延はトレードオフになる。本要件では **VPN 併用は選択制とし、既定では平文伝送であることを利用ガイドに明記する** `MUST`(ドキュメント化としての MUST)
  - 親プロジェクトとの関係(重要な既定方針の相違): 親 `requirements.md` §3 / `architecture.md` は「通話はE2EEを標準にする」ことを、VPSが借り物でありVPS侵害時にも内容が漏れない状態を保つためと明記している。本系統(selfmatrix-hires)が運ぶハイレゾ音声も同じくVPSを経由し、同じ「会話内容」を運ぶにもかかわらず、本系統は既定で平文とする。これは**親のセキュリティ原則を意図的に適用対象外とする別系統としての判断**であり、読者が「本体はE2EEだから安全」という認識のままハイレゾ経路(平文UDP)に機密性の高い会話を流すと、VPS侵害時に本体側では守られるはずだった内容が漏洩し得る。この乖離を利用ガイドで明確に注意喚起する
  - 検証方法: README/利用ガイドに「UDP 音声は暗号化されない」旨、VPN 併用時のレイテンシ影響、および「本系統は本体の通話E2EE方針とは別系統であり、E2EEの適用対象外である」旨が明記されていること
- hub 同時接続数の運用者による明示的な上限設定は、コマンドラインオプションとしては存在しない(ソースコード上 `gMaxThreads=1024`、UDP ポート割当は 61002-62000 の範囲というソフトウェア上限のみ) `既知の制約として明記`
  - 対応方針: 現行実装(`provision.sh --max-clients`)はアプリケーション層の上限ではなく、ufw の UDP ポート開放範囲を絞ることで実質的な上限を作る方式であり、この方式を継続する `MUST`

### 4.3 録音

用途として「収録・録音素材」を選択したため(§1.1)、実現方式を比較のうえ推奨を示す。**方式A・方式Bとも現時点(Stage 0)では未実装であり、`provision.sh`/README に録音機能のコード・記述は存在しない。Stage 1 は README 上で方式Bの利用手順を案内文書として追加するのみであり、hub側録音ツール(方式A)のコード化は行わない。**

#### 方式比較

| 方式 | 概要 | 長所 | 短所 |
|---|---|---|---|
| A: hub 側録音 | VPS 上で `jack_capture` 等により `send_N`(個別生音)/`receive_N`(参加者視点ミックス)を録音 | 参加者側の操作不要。個別トラックとミックスの両方を選んで録音できる(JACK ポート単位で区別可能なことを `Patcher.cpp`/`JackAudioInterface.cpp` で確認済み) | VPS はステートレス運用が前提(親要件 §6)であり録音ファイルの永続化と相性が悪い。384kHz/32bit float のステレオ1トラックは 384000×4byte×2ch ≈ 3.07MB/s ≈ 11.1GB/h で、10人分の個別トラックでは1時間あたり約110GB(ミックスのみでも約11GB/h)に達し、一般的な VPS ディスク(25〜100GB級)を圧迫する。JackTrip 公式録音スクリプトは 48kHz 固定前提で改修が必要。192kHz でのjack_capture 倍速録音バグ報告あり(384kHz での再現は未確認) |
| B: クライアント側録音(double-ender) | 各参加者がローカルで自分の送受信音声を個別に録音し、セッション後に持ち寄って合成 | VPS に録音データが一切残らずステートレス原則と矛盾しない。ポッドキャスト業界で標準的な手法。通信品質に依存しない高音質な個別トラックが得られる | 参加者ごとに録音操作・ファイル授受の手間が発生。全員の協力が前提になる |

- **推奨: 方式 B(クライアント側 double-ender)を基本とする** `SHOULD`
  - 理由: VPS ステートレス原則(親要件 §6「VPS には永続データを置かない」)と正面から整合する。384kHz 運用時の容量問題(§7)を hub 側に持ち込まずに済む
- hub 側録音(方式 A)は「バックアップ/簡易確認用途」に限定し、採用する場合は次の条件を満たすことを要件とする `LATER`
  - 容量を抑えるため 192kHz/24bit ステレオミックスのみを対象とし、384kHz/32bit float の hub 録音は対象外とする
  - 録音ファイルは短時間(セグメント単位)で VPS 外へ転送し、転送完了後は VPS 側ファイルを確実に削除する(ステートレス原則を破らないための運用手順として必須化)
  - 検証方法: セグメント転送 → 削除のサイクルが自動化スクリプトで完結し、VPS 再起動後にファイルが残っていないこと
- 録音同意の取得を運用ルールとして明記する `MUST`
  - 根拠: 一方当事者同意で法的には足りるとされるが(参考情報。法令原文レベルの確認は未了、§9 未解決課題)、複数参加者が共同利用するシステムである以上、セッション開始時の明示同意フローを運用ガイドに含める
  - 検証方法: 運用ガイドに「録音するセッションでは開始前に全参加者へ録音の旨を告知し同意を確認する手順(口頭またはチャットでの確認)」と「同意しない参加者がいる場合の対応(録音しない、または当該参加者の退出後に録音開始)」が明記されていること

### 4.4 半自動ランチャー

- ネイティブアプリ前提(§2)は維持しつつ、接続設定の配布とワンコマンド/ワンクリック接続をこのリポジトリで提供する `MUST`(Stage 2、§7。Stage 1 のスコープには含まない)
  - リポジトリ内完結の方針: ランチャーはこのリポジトリ内のシェルスクリプト/バッチファイルとして実装し、外部配布サービスや本体(SelfMatrix)リポジトリには依存しない。§0 の独立拡張方針(本体非結合)と整合させる
  - 対象 OS: Windows / macOS / Linux(現行 README のインストール手順に準拠。スマートフォンは非対応)
  - 自動化する範囲: 接続コマンド(ホスト名・サンプルレート・ビット深度・チャンネル数・認証情報)の組み立てと実行を、設定ファイル(例: 運用者が配布する `.json`/`.env` 相当)からワンコマンドで行えるようにする。JackTrip 自体には ini/json 設定ファイルや環境変数によるオプション指定機構が存在しないことをソースコード(`Settings.cpp`)で確認済みのため、ランチャー側でシェルスクリプト/バッチファイルがコマンドライン引数を組み立てる方式を採る。なお `-T, --srate` は rtaudio モード(`-R`)専用オプションであり(§6)、クライアント側は `-R` モードでの起動を前提にコマンドを組み立てる
  - 自動化しない範囲: オーディオインターフェースの選定・OS 側のサンプルレート設定・ASIO ドライバの導入は参加者の手動作業として残す(ハードウェア依存のため自動化の対象外)
  - `jacktrip://` ディープリンクは商用 Virtual Studio Mode 専用の機能であり、本要件が前提とする Classic mode(自前 hub、CLI)では機能しないことをソースコード調査(`UdpHubListener.h/.cpp` に URI スキーム処理なし)で確認済み。ランチャーはディープリンクに頼らず、接続情報(ホスト名・認証情報)をテキストで配布し、ローカルのランチャーがそれを読んで起動する方式とする
  - 検証方法: 参加者が「設定ファイルを受け取る」→「ランチャーを実行する」の 2 ステップで接続完了すること(コマンドライン引数を手打ちしない)

### 4.5 運用

- 人数上限は最終的に本体の通話人数上限(10人前後、親要件 §3)に合わせることを目指し、段階的に引き上げる(方針。タグなし — 各段階で実際に満たすべき人数上限は §7 の各 Stage exit criteria に従う)
- 二重再生防止は運用ルール(自分の出力ミュートまたは本体からの退出)で対応する。クライアント側自動ミュートの実装は行わない `OUT`
  - 理由: 本体との非結合方針(§2)により、本体側に検知・制御ロジックを持たせない
- 自動帯域/人数ガード(接続時に帯域超過を検知して自動的に接続を絞る機構)は実装しない `LATER`
  - 理由: ユーザー決定により非選択機能。現時点では運用者が `--max-clients` を手動で設定し、ufw のポート範囲で実質的な上限を作る方式(現行実装)を継続する。将来的に人数が本格的に10人規模に近づいた場合は、監視・自動制限の必要性を再評価する
- 複数同時セッション(hub の複数インスタンス並走)は実装しない `LATER`
  - 理由: ユーザー決定により非選択機能。現行実装は「同時1セッションのみ」を前提とする。`--bind-port` を変えた手動増設は可能(現行 README §5.2 記載)だが、これは回避策であり正式な複数セッション機能ではない

## 5. 非機能要件

### 5.1 遅延

- 会話が成立する片道遅延(目標 150ms 以内)を最優先とする `MUST`
  - 実測: ネットワーク RTT 4.4ms(自宅⇔VPS)、JACK 周期 256/192000 ≈ 1.33ms、hub キュー `-q 8` ≈ 10.7ms の合計で片道 20ms 未満(hires-spike.md §2)。要件(150ms)に対し実測は一桁近く余裕がある
  - 384kHz/32bit float 移行後も、周期を小さく保てば(§5.2)同等のレイテンシ水準を維持できる見込みだが、実機での再測定を Stage 3 の exit criteria とする(§7)
  - 検証方法: 実オーディオデバイスでのループバック録音による遅延実測(README §5 記載の手順を各段階で再実施)

### 5.2 帯域

計算式: ビットレート = サンプルレート × ビット深度 × チャンネル数。UDP パケットは `DefaultHeader(16byte) + JACK周期 × チャンネル数 × バイト数` で構成され、Ethernet MTU(1500byte)制約から実効ペイロード上限は 1472byte(IP+UDPヘッダ差引後)。

| 段階 | 1人あたり片方向理論値 | 実測/備考 | MTU 制約下の周期上限 |
|---|---|---|---|
| 192kHz/24bit ステレオ | 192000×24bit×2ch = 9,216,000bps ≈ 9.22Mbps | 実測 9.75〜9.9Mbps(ヘッダ込み、周期128〜256) | 周期 128(784byte)以下を推奨。周期256(1552byte)はMTU超過でIPフラグメント化し、WSL2等のNAT環境で接続不可になることを実機確認済み |
| 384kHz/32bit float ステレオ | 384000×32bit×2ch = 24,576,000bps ≈ 24.58Mbps | 未実測(Stage 3 で実測予定) | ペイロード = 16 + 周期×8。周期182サンプルが1472byte上限の境界のため、2の冪で安全なのは周期128(1040byte)以下。周期256(2064byte)はMTU超過 |

人数×段階での hub 合計帯域(片方向。双方向は概ね2倍):

| 人数 | 192kHz/24bit(実測ベース) | 384kHz/32bit float(理論値、圧縮なし) | 384kHz/32bit float(圧縮あり、仮に50%削減) |
|---|---|---|---|
| 2人 | ≈ 19.5Mbps | ≈ 49.2Mbps | ≈ 24.6Mbps |
| 6人(Stage 1 既定上限) | ≈ 58.5Mbps | ≈ 147.5Mbps | ≈ 73.7Mbps |
| 10人(最終目標) | ≈ 97.5Mbps | ≈ 245.8Mbps | ≈ 122.9Mbps |

(表は片方向合計。双方向では概算 2 倍になる点に注意。hires-spike.md の実測表は rx+tx 合計で記載しているため単純比較時は注意)

- **384kHz/32bit float × 10人は生データで片方向約 245.8Mbps に達し、可逆圧縮を仮に 50% 効かせても約 122.9Mbps で、現行 100Mbps 級 VPS 回線には収まらない** `既知の制約として明記`
  - 実測(2026-07-06、合成素材・blocksize=128、[docs/stage2-compression-gateway.md](stage2-compression-gateway.md) §4)では圧縮率は 48〜93% と素材依存で、表の「50%削減」は speech 系素材の楽観値。**圧縮は「効くこともある」補助最適化と位置づけ、収容人数の計画は非圧縮値を基準に立てる**
  - 対応方針: この問題は単一の解決策では裁かない。以下の組み合わせで段階的に対応する(§7 ロードマップに反映)
    1. 回線増強(上位プランへの変更)
    2. 段階的な人数制限(384kHz/32bit float は当面 2〜6人規模までとし、10人規模は 192kHz/24bit または圧縮ありに限定する)
    3. 可逆圧縮の実装(§4.1)
  - 検証方法: Stage 3 到達時点の VPS 回線プランと圧縮実測値をもとに、実際に何人まで収容できるかを再計算し、収容できない人数は Stage 3 の対象外として明記する

### 5.3 RAM/CPU

- hub の RAM 消費は接続人数に応じて増加する `既知の制約として明記`
  - 実測: 2クライアント接続時で hub オーディオスタック合計 ≈400MB(1GB VPS の空きが 273MB まで低下)。10人規模を日常的に回すなら 2GB クラスの VPS を前提とする(現行 README §3.3 記載)
- hub の CPU 負荷: 2クライアント時で jacktrip ≈28% + jackd ≈6%(1コア比、hires-spike.md 実測)。384kHz/32bit float・圧縮ゲートウェイ導入後の負荷は未実測のため、Stage 2/3 の exit criteria に実測を含める

### 5.4 コスト

- Stage 1(192kHz/24bit、6人規模)は現行 VPS プラン(1GB RAM、100Mbps級)で運用可能 `MUST`(実測ベース)
- Stage 2(圧縮導入・ランチャー)は RAM 2GB クラスへの引き上げを要件に含める `SHOULD`(§5.3 根拠)
- Stage 3(384kHz/32bit float・10人規模)は回線増強を伴う可能性が高く、VPS プラン変更のコスト増を運用者が受け入れる前提で進める `LATER`
  - コスト上限は本ドキュメント時点で未確定であり、具体的な月額上限額は §9 未解決課題とする。Stage 3 exit criteria(§7)において、回線増強の見積り額が運用者の許容額を超える場合は、10人規模を諦めて対応可能な人数規模まで Stage 3 のスコープを縮小する(§5.2 の人数×帯域試算に基づき、許容予算内に収まる人数を上限として再設定する)

### 5.5 運用負荷

- provision スクリプトの再実行安全性(既存ユーザー・証明書の非上書き)を維持する `MUST`(現行実装済み)
- オンデマンド起動(RAM 節約のため常時起動しない)を維持する `MUST`(現行実装済み)
- JackTrip・依存パッケージのバージョンを `provision.sh` 内で固定し、更新は明示的な検証を経て行う `SHOULD`
  - 根拠: 親要件 §8 は非機能要件として「再現性: イメージタグ固定と Renovate 等による更新PR運用」を MUST としているが、本系統では JackTrip/依存パッケージのバージョン固定・セキュリティアップデート追従方針が未定義だった。UDP 音声ペイロードが平文であること(§4.2)を踏まえると、脆弱性修正版への追従方針の不在は運用リスクである
  - 検証方法: `provision.sh` 内でバージョンが明示的に固定されていること、更新時の検証手順が README/運用ガイドに記載されていること

## 6. 制約と既知の事実(サマリー)

以下は調査・実測で確認済みの事実。要件の前提として扱う。

- JackTrip `-b 32` は int32 変換ではなく、JACK 内部の `sample_t`(float32)を `memcpy` でそのまま伝送する本物の float32 伝送である(`AudioInterface.cpp` 確認済み)
- JackTrip の UDP 音声ペイロードは暗号化されない(`UdpDataProtocol.cpp` に暗号化処理なし)。`-A` 認証は TCP ハンドシェイクのみを保護する
- JackTrip には ini/json 設定ファイル・環境変数によるオプション指定機構が存在しない(`Settings.cpp` 確認済み。コマンドライン引数のみ)
- `jacktrip://` ディープリンクは商用 Virtual Studio Mode 専用で、Classic mode(自前 hub)では機能しない
- hub 同時接続数を運用者が明示的にコマンドラインで制限する機能は存在せず、ufw のポート範囲制限で代替する必要がある
- UDP パケットサイズは MTU(1500byte)を超えるとフラグメント化し、WSL2 の NAT 等一部環境では接続不能になる(実機確認済み)。JACK 周期は 128 以下を安全側の既定値とする
- サンプルレート列挙型 `samplingRateT` は `SR22〜SR192 + UNDEF` で 384kHz の列挙子は存在しない(v2.2.2/v3.0.0/main で同一、2026-07-06 ソース確認済み)。384kHz は `UNDEF` として扱われるが、ヘッダ検証は列挙値の一致比較のみのため両端を揃えれば接続は成立する見込み。さらに `getSampleRateType()` の SR192 判定タイポ(`19200`、`AudioInterface.cpp:799`)により**現行 192kHz 運用も UNDEF 扱いで動作しており、スパイク PASS は UNDEF モードでの実績**である(§3.1)。UNDEF モードではレート不一致検出が機能しない点に注意
- `-T, --srate` オプションは rtaudio モード(`-R`)専用であり、JACK バックエンド使用時はサンプルレート指定オプションが存在せず jackd が確立したレートに追従する。hub 側は jackd がレートを規定し、クライアント側は `-R` モードで `-T` 指定するという非対称な構成になる
- RtAudio バックエンドには `checkSampleRate()`/`getClosestSampleRate()` という機構があり、要求レートにデバイスが対応しない場合、libsamplerate が利用可能ならサンプルレート変換、なければ実行時例外となる。libsamplerate 利用可能な環境では、指定レートと異なるレートへ黙って変換されても「エラーなく接続成立」に見えてしまう点に注意(§3.1 exit criteria 参照)
- JackTrip 本体のライセンスは AGPL-3.0 系と見られる(要再確認、§9)。圧縮ゲートウェイ(§4.1)を別プロセスとして実装する場合はライセンス条件と抵触しない見込み
- 384kHz/32bit float 対応の市販オーディオインターフェースはハイエンド機に限られ、一般的なマイクの周波数特性(20Hz〜20kHz程度)は 384kHz サンプリングの理論上の恩恵をそのまま活かせない
- WavPack は 32bit float PCM をネイティブ・可逆に扱える。実測(2026-07-06、合成素材、WavPack 5.9.0): blocksize=128 で圧縮率 48〜93%(素材依存)、エンコード速度は最悪でも実時間の 12.8 倍でエンコード単体の CPU は問題にならない([docs/stage2-compression-gateway.md](stage2-compression-gateway.md) §4/§6)
- hub の JACK グラフ上で `send_N`(個別生音)と `receive_N`(参加者視点ミックス)はポート単位で区別して録音対象にできる(`Patcher.cpp`/`JackAudioInterface.cpp` 確認済み)

## 7. 段階的ロードマップ

各 Stage は前 Stage の exit criteria を満たしてから着手する。

### Stage 0(現状、実装済み)

- 内容: 192kHz/24bit 非圧縮 PCM、hub mode、`-p 2`(client fan out/in)、認証(`-A`)、provision.sh による自動構築、既定上限6人
- exit criteria: 満たしている(現行 README/provision.sh がそのまま該当)

### Stage 1(初期リリース)

- 内容: Stage 0 を土台に、利用ガイド・トラブルシュートを本ドキュメントの要件水準に合わせて整備。UDP 非暗号化の明記、録音は方式 B(クライアント側)を案内として追加
- exit criteria:
  - 6人規模での接続安定性を実運用(聴感テストではなく実セッション)で複数回確認する
  - README に「UDP 音声は暗号化されない」「本系統は本体の通話E2EE方針の対象外である」「二重再生防止は運用ルール」「録音は各自ローカルで(double-ender)」が明記されている
  - 録音関連の実装(方式A・方式Bとも)は行わない。方式Bは README上の案内(文書)追加のみに留める(§4.3)

### Stage 2(圧縮 + 半自動ランチャー)

- 内容: WavPack ベースの圧縮ゲートウェイ PoC(§4.1)、半自動ランチャー(§4.4)の実装
- exit criteria:
  - JackTrip の UDP ペイロード構造と圧縮データの互換性を検証済み(挟み込みが技術的に成立するか、実質的に別トランスポートの自作が必要かを判定し、設計を確定させている)— **設計レベルで達成(2026-07-06、[docs/stage2-compression-gateway.md](stage2-compression-gateway.md): 案A不成立・案B透過トンネル型を採用)**。実装レベルの裏付けは PoC 段階1で行う
  - 圧縮ゲートウェイが 192kHz/24bit で片道 150ms 以内・実測圧縮率を計測済み
  - ランチャーで「設定ファイル受け取り→実行」の2ステップ接続が Windows/macOS/Linux それぞれで動作確認済み。ランチャーはこのリポジトリ内のスクリプトのみで完結し、外部サービス・本体リポジトリに依存していないことを確認(2026-07-06 時点: `launcher/` 実装済み・Windows で dry-run 検証済み。macOS/Linux 実機確認と実接続での確認が残)
  - VPS を 2GB クラスに引き上げ、RAM 実測が許容範囲内であることを確認

### Stage 3(384kHz/32bit float 対応 + 人数拡大)

- 内容: 384kHz/32bit float の実機検証、圧縮ゲートウェイの 384kHz 対応、人数を段階的に 6→10人へ拡大
- exit criteria:
  - 前提知識(2026-07-06 確定済み、§3.1): 列挙型に `SR384` は存在せず、384kHz は UNDEF モードでの動作になる(現行 192kHz 運用と同じコードパス)。Stage 3 は UNDEF モードでの実機検証として計画する
  - 実オーディオデバイスで 384kHz/32bit float の接続成立・XRUN なし・遅延実測が 150ms 以内であることを確認(dummy driver ではなく実機)
  - 実際に 384kHz のまま伝送されていることを実測帯域(1人あたり片方向 ≈24.6Mbps、§5.2)で確認する(ヘッダの `SamplingRate` は UNDEF 固定のため確認手段にならない)。libsamplerate による黙示的なリサンプリングで見かけ上「接続成立・XRUNなし」になっているだけの状態ではないことを切り分ける(§6)
  - 上流への `SR384` 追加 + `19200` タイポ修正パッチの contribution(またはフォークパッチ運用)の要否を判断し、結果を決定記録に残す
  - 帯域試算(§5.2)を踏まえ、実際の VPS 回線プランで何人まで収容できるかを再計算し、10人到達に回線増強が必要であればその見積りを提示する。見積り額が §5.4 で定める許容予算の上限を超える場合は、10人規模を諦め、予算内に収まる人数規模まで Stage 3 のスコープを縮小する
  - 384kHz 非対応の参加者へのフォールバック手順(192kHz への hub 建て直し)が Stage 1 の 192kHz/48kHz 切替と同様に運用可能であることを確認

Stage 3 の「10人」到達は回線増強が前提になる可能性が高いことを、この時点で明示しておく(§5.2)。回線増強を行わない場合、384kHz/32bit float は少人数(2〜6人)規模の上位互換オプションに留める。

## 8. 明示的 OUT・当面対象外

OUT(対象外と決めたもの)と LATER(将来対応。§4/§5 の各タグが正)をまとめて一覧化する。

| 項目 | 理由 |
|---|---|
| 音楽鑑賞・リスニング会 | ユーザー決定により非選択。BGM共有・同期再生等は本要件のスコープ外(OUT) |
| 楽器セッション・合奏 | ユーザー決定により非選択に加え、遅延要件が会話用途と一桁異なる別領域(§1.2)(OUT) |
| 複数同時セッション | ユーザー決定により非選択機能。`--bind-port` 変更による手動増設は可能だが正式機能化はしない(LATER、§4.5) |
| hub 側録音のフル解禁(384kHz含む全形式) | VPS ステートレス原則と容量試算(§4.3、10人分の個別トラックは384kHz/32bit floatで1時間約110GB)から、常設のhub側フル録音は採用しない。限定条件付きでLATER(§4.3) |
| 自動帯域/人数ガード | ユーザー決定により非選択機能。運用者の手動設定(`--max-clients`)とufwのポート制限で代替(LATER、§4.5) |
| クライアント側自動ミュート・二重再生の自動検知 | 本体との非結合方針により、本体側に検知ロジックを持たせない(親要件§9の踏襲)(OUT) |
| モバイルクライアント対応 | 現行実装がスマートフォン非対応。JackTrip自体もモバイル公式サポートが薄く、対象外(OUT) |
| 映像とのリップシンク保証 | 音声主体の設計方針(§2)(OUT) |

## 9. 未解決課題と検証計画

- ~~384kHz サンプルレート列挙型の矛盾~~(**解消済み 2026-07-06**、§3.1): 列挙型は `SR192` 打ち止めで 384kHz は UNDEF 扱いと確定。ただし現行 192kHz 運用も SR192 判定タイポにより UNDEF 扱いで動作しており、UNDEF モードの実用性は実証済み。残る課題は次の2点 — (1) **UNDEF モードでの 384kHz 実機検証**(Stage 3)、(2) **上流への `SR384` 追加 + `19200` タイポ修正の contribution 要否判断**(タイポ修正だけが上流に取り込まれた場合、192kHz が正しく SR192 になる一方 384kHz は UNDEF のままとなり、両者のヘッダ値が不一致になって接続拒否される、という将来の挙動変化にも注意が必要)
- ~~圧縮ゲートウェイの実現可能性~~(**設計レベルで解消 2026-07-06**、[docs/stage2-compression-gateway.md](stage2-compression-gateway.md)): 案B(透過トンネル型プロキシ、JackTrip 無改造)で成立と判断。残: PoC 段階1/2 — 実装、ストリーミング実処理での追加遅延、VPS 実機 CPU/RAM(10ストリーム時の余裕は現時点では断定不可、設計文書 §6)、実マイク素材での圧縮率、TCP ハンドシェイク覗き見の頑健性(JackTrip バージョン追従リスク)
- **JackTrip のライセンス**: AGPL-3.0系と見られるが正式な確認が済んでいない。圧縮ゲートウェイを別プロセスとして実装する場合の配布条件への影響を確認する必要がある(§4.1/§6)
- **Stage 3 のコスト上限**: 回線増強の許容予算(月額上限)が未確定。Stage 3 exit criteria の判断基準として、Stage 3 着手前に運用者の許容額を確定しておく必要がある(§5.4/§7)
- **384kHz 実機検証**: dummy driver ではなく実オーディオインターフェース(RME ADI-2 Pro 系等)での 384kHz/32bit float 接続成立・XRUN 有無・遅延実測が未実施。Stage 3 着手前に必須
- **10人同時接続の実測**: 現行の帯域・RAM試算はすべて2クライアントからの線形外挿であり、10人同時の実測は未実施(レート制限・マシン分散の再設計が必要)
- **圧縮方式の PoC**: 合成素材での圧縮率・エンコード速度は実測済み(2026-07-06、設計文書 §4)。残: ストリーミング実装(libwavpack)での追加遅延、VPS 実機での CPU/RAM、実マイク素材での圧縮率。FLAC を使う場合の float→int変換の音質影響評価も未着手
- **聴感 ABX テスト**: 「会話の高音質化」の効果を裏付けるための、48kHz/Opus vs 192kHz/24bit(および384kHz到達後はそれも含めた)聴感比較テストが未実施。効果が確認できない場合、対応目標の見直しを検討する
- **jack_capture の384kHz動作**: 192kHzでの倍速録音バグ報告(GitHub Issue)があり、384kHzでの再現有無が未確認。hub側録音(方式A、LATER)を実装する場合は事前検証が必要
- **libsndfile の FLAC 変換時の不具合**: 24bit float書き込み時のノイズ報告があり、現行バージョンでの修正状況が未確認。FLACを圧縮方式として採用する場合は要再検証
- **録音同意の法的要件**: 日本法上は一方当事者同意で足りるとされる情報があるが、法令原文レベルの確認は未了。個人情報保護法上の取り扱い(要配慮個人情報該当性等)も含め、法務確認を推奨
- **RedNet 1 等の384kHz対応の実態**: 一部業務用インターフェースの「384kHz対応」表記が特定モードに限定される可能性があり、一次資料での裏取りが不十分

## 11. 決定記録(2026-07)

- ハイレゾ系統を本体分離済みの独立拡張として、目的からゼロベースで要件を再定義した(2026-07-06)
- 用途を「会話の高音質化」「収録・録音素材」の2本に確定し、「音楽鑑賞・リスニング会」「楽器セッション・合奏」を理由付きで明示的 OUT とした
- 音質の対応目標を 384kHz/32bit float に格上げした(親要件では LATER)。ただし実機未検証のため MUST ではなく SHOULD とし、Stage 3 で実機検証を経てから確定する。2026-07-06 のソース確認で列挙型は `SR192` 打ち止めと確定(384kHz は UNDEF 扱い)したが、現行 192kHz 運用も SR192 判定タイポにより UNDEF 扱いで動作しておりスパイク PASS はその実績であるため、Stage 3 は UNDEF モードでの実機検証+上流改修要否判断として計画する(§3.1/§9)
- 人数上限は最終的に本体の通話人数上限(10人前後)に合わせる方針としつつ、帯域試算(§5.2)を踏まえ段階的に引き上げるロードマップ(§7)を採用した。回線増強コストが運用者の許容予算を超える場合は人数規模を縮小する(§5.4)
- 録音方式はクライアント側 double-ender を推奨とし、hub側録音はVPSステートレス原則との整合を理由に限定条件付きのLATERとした。方式A・Bとも本ドキュメント時点では未実装であり、Stage 1では方式Bの文書案内のみを追加する(§4.3)
- 可逆圧縮伝送は WavPack を第一候補として SHOULD で採用し、複数同時セッション・自動帯域/人数ガードは非選択機能として LATER/OUT に振り分けた。ただし「JackTripの前後に挟む」という実現方式は要検証の設計仮説であり、Stage 2 PoCでプロトコル互換性を検証してから確定する(§4.1/§9)
- 半自動ランチャーを新規要件として追加し、ネイティブアプリ前提を維持したまま接続設定配布とワンコマンド接続をこのリポジトリの責務とした(Stage 2、リポジトリ内完結・外部非依存)
- 本系統は本体の通話E2EE方針の適用対象外であり、別系統として意図的に平文伝送を許容することを明記した(§0/§4.2)
- Stage 2 の圧縮ゲートウェイは案B(透過トンネル型プロキシ、JackTrip 無改造)に設計確定した(2026-07-06、docs/stage2-compression-gateway.md)。WavPack 実測により §5.2 の「圧縮50%」仮定は素材依存の楽観値と判明したため、収容人数の計画は非圧縮値基準に修正した。半自動ランチャー(launcher/)を実装し、Windows での dry-run 検証まで完了した
